# Tidying Commit 가이드 (A+ Skill 팀 기준)

## 1) 정의
`tidying commit`은 **기능 추가보다 코드 품질/안전성/가독성 정리**에 초점을 둔 작은 커밋입니다.

핵심은:
- 작게 나누고
- 안전하게 검증하고
- 꾸준히 누적하는 것

---

## 2) 어떤 변경이 tidying commit인가?

### 포함 (권장)
- 변수/함수명 가독성 개선
- 중복 코드 정리(동작 동일)
- 로그/에러 메시지 구조화 및 마스킹
- 타입/주석/문서 정리
- 테스트 보강(회귀 방지)
- 경계값/입력 검증 강화
- 설정값/상수 정리

### 제외 (다른 커밋으로 분리)
- 신규 기능 추가(feat)
- API 계약 변경/DB 스키마 변경
- 동작 의미가 크게 바뀌는 로직 변경
- 다수 기능을 한 번에 섞은 대형 수정

---

## 3) 커밋 메시지 규칙
항상 아래 형식 사용:

`tidying commit: <무엇을 정리했는지 한 줄>`

예시:
- `tidying commit: sanitize delivery mode log fields`
- `tidying commit: tighten retry-after parsing and clamp`
- `tidying commit: clarify fallback output messaging`

---

## 4) 크기 규칙 (권장)
- 파일 수: 가능하면 1~5개
- 변경 라인: 가능하면 200줄 내
- 목적: 한 커밋 = 한 의도

---

## 5) 검증 규칙 (필수)
커밋 전 최소:

```bash
npm test
npm run typecheck
npm run build
```

우리 작업 흐름에서는 추가로:
- 리뷰 에이전트
- 비판적 사고 에이전트
- 검증 에이전트

**3중 루프 통과 시 커밋/푸시**

---

## 6) 실무 운영 팁
- “완벽한 큰 개선”보다 “안전한 작은 개선”을 우선
- 리스크가 큰 변경은 실험 브랜치에서 검증 후 병합
- 로그/리포트/외부 전송 경로는 민감정보 노출을 먼저 점검
- tidyings는 기능 속도를 늦추는 게 아니라, 장기적으로 속도를 올리는 투자

---

## 7) A+ Skill 전용 체크리스트
커밋 전에 확인:
- [ ] 민감정보(토큰/ID/경로/원문 에러) 노출 없음
- [ ] fallback/degraded 동작이 리포트에서 안전하게 표현됨
- [ ] 429/5xx/4xx 재시도 정책 일관성 유지
- [ ] 기존 테스트 회귀 없음
- [ ] 커밋 메시지가 `tidying commit:` 규칙 준수
